# TianShu Current Product Definition

Status: current
Updated: 2026-07-15

## Product

TianShu is a personal cognitive operating system. It helps the creator
understand change, make decisions, act within explicit boundaries, learn from
outcomes, and preserve confirmed experience as long-term cognitive assets.

The product loop is:

`Understand -> Judge -> Act -> Feedback -> Consolidate -> Evolve`

AI augments the creator. It does not replace the creator. Identity changes,
long-term memory, major actions, and final acceptance remain under creator
control.

The current operator is TianShu's first real user and final product acceptor,
not the product identity. User-facing names and preferences are SQLite product
profile data. V1.1 remains single-user first without hard-coding that first
user into authority, sessions, APIs, or interface copy.

## Current stage

The current product stage is **V1.1 Minimum Real Intelligence Loop**. P1-P6 are
engineering capability stages and must not replace the product-stage label.

V1.1 is complete only when one real workflow demonstrates all of the following:

1. A real input is interpreted using confirmed creator context and evidence.
2. TianShu produces an explainable judgment and one useful next action.
3. The creator can accept, correct, reject, or defer the judgment.
4. Any execution stays inside a separately confirmed boundary.
5. The outcome and creator feedback are recorded.
6. A resulting experience candidate is explicitly reviewed.
7. A later similar judgment can cite the accepted experience and improve.

Automated samples, executor exit codes, and isolated demonstrations cannot by
themselves complete V1.1.

## Product surfaces

- **TianShu workspace:** the primary daily operating surface.
- **AgentHub:** an interaction client for Today, decisions, actions, and evidence.
- **Obsidian:** a creator reporting and readable mirror surface. Markdown is
  never machine-state authority.
- **Gateway:** the governed intake and decision boundary for every surface.
- **SQLite:** the only machine-state authority.

## Current priorities

1. Make the product loop visible and usable without reading JSON or SQLite.
2. Establish a stable workspace information architecture.
3. Implement governed proactivity with low interruption and clear suppression.
4. Close the feedback-to-evolution loop before adding more integrations.
5. Prove usefulness through repeated personal use.

## Explicit non-goals for V1.1

- Unattended control of business projects.
- Broad integration expansion before the core loop is stable.
- Automatic promotion of identity, long-term memory, or operating rules.
- A dashboard whose primary value is metrics rather than decisions.
- Replacing the runtime or state architecture without evidence.
