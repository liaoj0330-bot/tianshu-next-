# 天枢奈奈版系统补全与工作台重组方案

Status: proposal
Owner: 奈奈
Updated: 2026-07-15

## 1. 结论

天枢当前不是“没有系统”，而是底层治理能力、产品工作区、知识库目录和交互入口没有形成同一套语言。

最优方向不是继续增加文件夹，而是固定三层：

1. **SQLite 权威层**：保存正式状态、版本、权限、确认、执行、验证、结果和经验。
2. **天枢认知与治理层**：理解输入，读取奈奈的已确认上下文，形成可解释判断，管理确认边界，编排执行和独立验证。
3. **用户表面层**：天枢工作台、AgentHub 和 Obsidian 都只读取或提交受治理请求，不各自拥有一份事实。

奈奈每天只需要从一个入口开始。系统结构可以复杂，但使用路径必须保持：

```text
看今天需要注意什么
  -> 看天枢为什么这样判断
  -> 接受、纠正、拒绝或稍后处理
  -> 必要时另行确认执行边界
  -> 查看执行和独立验证证据
  -> 奈奈最终验收
  -> 决定是否把本次经验用于未来
```

## 2. 对四份外部文档的取舍

吴老师的文档最值得天枢学习的不是目录和技术栈，而是四类系统原则：

| 应继承的原则 | 奈奈版落地方式 |
| --- | --- |
| 创造者主权 | 奈奈是唯一目标拥有者、正式状态确认者、执行批准者、最终验收者和经验提升者 |
| 认知闭环 | 理解、判断、行动、反馈、沉淀、进化必须成为可见的状态链，而不是一段聊天 |
| 证据与边界 | 判断展示证据、推断、不确定性；执行必须二次确认；执行者不能自证完成 |
| 长期演化 | 结果先成为经验候选，经过奈奈确认、限定范围和版本后才能影响未来判断 |

以下内容不能直接照搬：

- 文档中的“创造者”角色必须完整替换为奈奈的真实目标、偏好、边界和项目关系，不能继承吴老师的身份上下文。
- 历史 V1 的 React、localStorage 或目录描述不能覆盖当前 Node.js、Gateway、SQLite 架构。
- 外部建议只能作为咨询来源，不能直接改变天枢正式状态。
- “会调用很多 Agent”不是产品价值。只有当任务确实需要跨工具、长时运行、并行或独立验证时才进入 AgentHub 执行链。

## 3. 我对奈奈需求的系统定义

天枢要解决的不是“保存更多知识”，而是六个连续问题：

1. **知道奈奈**：知道哪些是已确认事实、当前状态、长期原则、偏好和不可触碰边界，并区分历史版本。
2. **理解变化**：把新输入与当前状态、项目、关系和历史经验对照，识别真正变化，而不是机械归档。
3. **形成判断**：给出事实、推断、不确定性、证据、备选项、推荐行动和确认后果。
4. **受控行动**：先确认计划，再单独确认执行边界；没有授权就不启动执行。
5. **独立验收**：执行结果只是证据，独立验证通过后仍由奈奈最终决定是否完成。
6. **从结果进化**：接受纠正、反例和结果，形成有来源、有范围、有版本、可撤回的经验。

V1.1 的真正验收不是“功能都存在”，而是同类真实任务至少运行两次，第二次判断明确引用第一次由奈奈接受的经验，并且比第一次更适合奈奈。

## 4. AgentHub 为什么需要用

### 4.1 必要性

AgentHub 的必要性来自交互和执行的复杂度，不来自天枢缺少一个大脑。

AgentHub 应当承担：

- 接收自然语言、文件或设备入口提交的输入；
- 展示 Today、判断卡、计划草案、执行边界、运行状态和证据；
- 承接长任务的异步状态、取消、重试、恢复和通知；
- 在明确授权后，把任务交给合适的执行者和不同的验证者；
- 把每一步结果通过 Gateway 送回 SQLite。

AgentHub 不应当承担：

- 保存天枢的正式记忆或项目真相；
- 以 `agenthub` 身份替奈奈确认正式状态；
- 自己决定执行边界或默认启动 Agent；
- 把 Executor 的成功退出码当作目标完成；
- 把聊天记录当成可直接生效的长期经验。

因此，AgentHub 是**可替换的交互与执行驾驶舱**，不是天枢的大脑、记忆库或最终权威。即使 AgentHub 暂时不可用，SQLite 中的状态和 Gateway 的治理合同仍必须成立。

### 4.2 什么时候使用，什么时候不使用

| 输入类型 | 是否进入 AgentHub 执行链 | 原因 |
| --- | --- | --- |
| 查询已确认事实、项目状态、历史证据 | 否 | 直接读取权威读模型即可 |
| 请天枢分析、比较、判断或提出建议 | 通常否 | 先生成判断，不应因为“分析”就派发执行者 |
| 报告近况、纠正状态、提交经验 | 否 | 先形成候选并等待奈奈确认 |
| 修改文件、运行代码、跨工具收集证据 | 是 | 需要明确执行边界、状态追踪和证据 |
| 长时、并行、多步骤或可失败任务 | 是 | 需要异步编排、重试、恢复和取消 |
| 需要独立质量验证的交付 | 是 | 必须分离 Executor 和 Verifier |
| 对外发送消息、发布或修改外部系统 | 是，且高风险确认 | 外部影响不可仅凭聊天意图推定授权 |

### 4.3 当前 AgentHub 合同还缺什么

现有代码已经把 AgentHub 限定为 `interaction_channel`，这是正确的；但产品化前还需要：

- 创建者会话或签名动作，明确区分“奈奈在 AgentHub 中点击确认”和“AgentHub 自己确认”；
- 每次提交的 `idempotency_key`，避免网络重试产生重复 intake 或重复执行；
- 贯穿 intake、plan、task、run、verification 的 `correlation_id`；
- 完整的异步状态、取消、失败、重试和恢复合同；
- 判断卡与证据卡，而不是只显示 JSON；
- 明确的能力声明，让简单问题不误入多 Agent 执行链。

## 5. 为什么当前工作台显得乱

截图中的目录同时混入了四种不同概念：

| 当前目录 | 实际概念 | 问题 |
| --- | --- | --- |
| `00_入口导航` | 用户入口 | 与根目录的“从这里开始”重复 |
| `00_创造者模型` | 奈奈的跨域模型 | 被误当成普通存储文件夹 |
| `04_判断资产卡库` | 一类核心实体/视图 | 不应由用户手工维护为孤立卡库 |
| `06_项目记忆层` | 项目时间线和证据 | 与项目推进区分离后，同一项目出现多个真相 |
| `07_资产索引层` | 系统生成的索引 | 不应成为日常导航层 |
| `20_长期工作区` | 时间跨度 | 与内容分类维度不一致 |
| `30_项目推进区` | 项目生命周期视图 | 应与项目本体合并，用状态过滤 |
| `90_待处理` | 暂态队列 | Inbox 是状态，不是永久工作区 |
| `98_历史归档` | 历史过滤视图 | 正式历史应来自 SQLite 事件和版本，不靠手工搬文件 |

根因是把“导航、内容对象、技术层、生命周期和暂态队列”放在了同一级。目录命名本身没有错，但分类轴不一致，所以奈奈每次都要先判断东西应该放在哪一层。

## 6. 目标信息架构

### 6.1 三种结构必须分开

**产品分类工作区**保持现有正式合同：

`Today / Projects / Life / Relationships / Knowledge / Evolution / Activity`

`Inbox` 只表示尚未完成分类或等待奈奈确认的输入状态，不作为长期存放位置。

**跨工作区核心对象**包括：

- 奈奈模型；
- 判断；
- 目标与计划；
- 项目；
- 证据；
- 执行与验证；
- 经验及其反例。

这些对象可以出现在多个工作区视图中，但在 SQLite 中只保留一份正式记录。

**用户导航**不需要照抄数据库表或七个分类器名称。建议只保留以下可见结构：

```text
00_从这里开始_天枢工作台.md       # Today：唯一日常入口
10_奈奈/
  奈奈当前模型.md                 # 只展示已确认内容、来源和版本
  奈奈模型变更历史.md
20_项目/
  项目总览.md
  <项目名>/
    当前状态.md
    时间线.md
    判断与证据.md
30_生活与关系/
  生活.md                         # Life 读模型
  关系.md                         # Relationships 读模型
40_知识与判断/
  判断账本.md                     # 跨项目判断视图
  证据索引.md
  来源与有效性.md
50_进化/
  经验候选.md
  已确认经验.md
  反例与回滚.md
60_活动/
  执行中.md
  待验收.md
  失败与恢复.md
  审计记录.md
99_系统/
  使用边界.md
  同步与健康.md
```

这套结构中，Today 是根入口；奈奈模型和判断账本是重要产品表面，但不是 intake 分类工作区；Life 和 Relationships 保持底层独立，在导航上合并以减少噪音。

### 6.2 当前目录迁移关系

| 当前目录 | 目标位置 | 处理方式 |
| --- | --- | --- |
| `00_入口导航` | 根入口页 | 合并，消除双入口 |
| `00_创造者模型` | `10_奈奈` | 改为 SQLite 生成的确认模型与历史视图 |
| `04_判断资产卡库` | `40_知识与判断/判断账本.md` | 从手工卡片改为正式判断记录的投影 |
| `06_项目记忆层` | `20_项目/<项目>/时间线.md` | 合并到对应项目，保留来源和时间 |
| `07_资产索引层` | `40_知识与判断/证据索引.md` | 由知识索引自动生成 |
| `20_长期工作区` | 按内容分入项目、生活、关系、知识或进化 | 取消“长期”这一混合分类轴 |
| `30_项目推进区` | `20_项目` | 通过聚焦、维护、暂缓、完成等状态筛选 |
| `90_待处理` | 根入口页的待确认区 | 不保留永久文件夹 |
| `98_历史归档_平时不用` | `60_活动` 的历史过滤视图 | 不手工搬运正式状态 |

迁移必须是非破坏性的：先生成新镜像，做来源和内容哈希对照，奈奈验收后才停止展示旧目录。旧文件不直接导入正式状态。

## 7. 底层已经具备与仍需补全的能力

### 7.1 已经具备

- SQLite 作为唯一机器状态源；
- 奈奈、AgentHub、Executor、Verifier、Obsidian 的权限边界；
- intake 工作区分类、歧义确认、修正和版本历史；
- Today 读模型和统一确认队列；
- 项目画像、优先级、项目变化候选和当前状态；
- 目标、计划、执行边界、任务、运行、独立验证和最终验收；
- 动态状态候选和记忆候选；
- 知识实体、来源、证据、关系和索引；
- 持久化判断、奈奈反馈、结果确认、经验候选、经验激活和后续判断引用；
- 事件、连续性、问题和进化候选基础。

### 7.2 关键缺口

1. **七个工作区只有分类合同，没有七套完整读模型。** 当前真正成熟的是 Today；Projects、Life、Relationships、Knowledge、Evolution 和 Activity 尚未形成统一产品投影。
2. **判断闭环已经有最小持久化实现，但还没有成为统一产品读模型。** 当前 `judgments`、反馈、结果和经验引用已进入 SQLite；仍缺少判断列表/检索、与 Today 统一确认队列的完整连接、判断修订表达、忽略反馈、效果评价和面向奈奈的判断账本。
3. **经验已经具备身份、首版版本、适用范围和使用记录，但完整生命周期仍未完成。** 当前可创建并激活第一版经验，也能限制后续判断只引用已激活版本；仍缺少新版本提议、撤回/失效、结构化反例、使用结果评价和回滚操作。
4. **奈奈模型尚未形成单一可解释读模型。** 当前状态、项目画像和偏好信号分散，缺少“已确认事实 / 当前状态 / 原则边界 / 未决变化 / 来源版本”的统一视图。
5. **Obsidian 仍是概念边界，没有可重复生成的完整镜像管线。** 当前不能安全地直接整理旧目录并宣称完成。
6. **AgentHub 只完成了基础接入，不是完整认知驾驶舱。** 卡片、身份确认、异步状态、证据、幂等和恢复仍需实现。
7. **真实双循环证据缺失。** 自动化测试通过不能代替奈奈在两个相似真实任务中的接受、纠正和体验提升。

### 7.3 建议补全的正式合同

当前 judgment migration 已覆盖其中一部分。实施阶段不重复造表，而是先复用并补齐以下领域合同：

- `creator_model`：确认事实、当前状态、原则边界、来源、有效期和版本；
- `judgment`：保留现有问题、事实、推断、不确定性、证据、备选和建议，并补上下文快照、有效修订表达和检索投影；
- `judgment_feedback`：保留现有接受、纠正、拒绝、延后，并补“忽略”、理由规范和后续效果；
- `outcome_review`：保留现有结果与奈奈确认，并补预期差异和质量评价；
- `experience`：保留现有身份、首版、来源结果和适用范围，并补新版本、结构化反例、撤回、失效和回滚关系；
- `experience_usage`：保留后续判断引用记录，并补实际影响和结果评价；
- `surface_delivery`：镜像或 AgentHub 投影的版本、哈希和投递状态；
- `request_idempotency`：入口请求、身份、键、响应和执行关联。

## 8. 推荐实施顺序

现有个人试点计划把 Obsidian 放在反馈与进化之前。建议在奈奈确认后调整这两阶段的顺序：先把已经实现的最小判断和经验闭环产品化、补齐生命周期，再生成镜像。原因是如果先固化目录，会把尚未完整的读模型复制成新的长期结构。

### Phase 0：冻结基线与迁移清单

交付：当前测试基线、API/表/读模型清单、旧目录内容映射、受限仓库清单和非破坏性迁移策略。

验收：全量测试通过；没有读取或写入 Teacher PPT、069、070 或业务项目；没有从 Markdown 导入运行状态。

### Phase 1：统一读模型骨架

交付：Today、Projects、Life、Relationships、Knowledge、Evolution、Activity 的统一只读投影；奈奈模型和判断账本作为跨工作区投影；统一来源、版本、确认状态和空状态合同。

验收：每个读模型都有可执行测试；读取不改变 SQLite；重启前后等价；Inbox 只以未决状态出现。

### Phase 2：判断与经验闭环产品化

交付：在现有最小闭环上增加判断账本与检索、五类奈奈反馈、结果复盘、经验新版本、结构化反例、撤回、回滚和经验使用效果披露，并接入统一确认队列。

验收：拒绝或被反例推翻的经验不能影响后续判断；第二次判断可以明确引用已接受经验；所有状态转换由权限测试覆盖。

### Phase 3：非破坏性 Obsidian 新镜像

交付：按第 6 节生成全新目录；来源哈希、幂等 intake、可重建镜像、旧目录到新目录的对照报告。

验收：重复报告不重复写状态；修改镜像不改变 SQLite；删除镜像后可等价重建；奈奈确认新导航后再隐藏旧导航。

### Phase 4：AgentHub 认知驾驶舱

交付：Today、Decision、Action、Evolution 四个主视图；创建者会话；判断和确认卡；异步运行、取消、重试、恢复和证据链；移动端与桌面端。

验收：简单问题不派发 Agent；计划确认不等于执行确认；Executor 与 Verifier 不同；AgentHub 不能以自身身份确认；奈奈无需查看 JSON 或终端即可完成闭环。

### Phase 5：真实奈奈双循环

交付：选一个低外部风险的真实任务连续运行两次；第一次完整记录判断、反馈、结果和经验，第二次披露经验引用与改进。

验收：奈奈最终接受；未经授权的外部动作数量为零；第二次在决策时间、重复解释或建议适配度中至少有一项可验证改善。

### Phase 6：有限主动性与长期试点

交付：去重、冷却、拒绝抑制、延后、安静时段、每日打扰上限和项目变化触发；三天工程回放、十四天奈奈使用，再决定是否进入三十天试点。

验收：不回复永远不等于同意；所有主动提醒和抑制都有 SQLite 证据；奈奈决定是否值得扩大集成。

## 9. 实施优先级

按风险和价值排序：

1. 权威状态和奈奈权限不能被任何界面绕过；
2. 判断与经验必须成为可追踪实体；
3. 七个工作区和跨工作区对象必须形成一致读模型；
4. Obsidian 生成新镜像并完成双轨验收；
5. AgentHub 补齐交互、身份、异步和证据合同；
6. 完成真实双循环后再扩大主动性和外部集成；
7. 视觉优化跟随认知阶段和状态合同，不先做装饰性大改。

## 10. 奈奈需要确认的架构决策

进入代码与工作台重组前，只需要确认以下整体方向：

- 七个工作区继续作为底层分类合同；
- 用户导航采用第 6 节的简化结构；
