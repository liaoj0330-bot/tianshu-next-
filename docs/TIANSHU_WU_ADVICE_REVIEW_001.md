# 天枢对吴老师四份文档的深度审议与奈奈适配方案 001

状态：审议候选，不代表奈奈已经作出最终决定
审议对象：TS-HO-02、TS-HO-03、TS-HO-05、TS-HO-06
唯一目标拥有者与最终验收者：奈奈
机器状态唯一真相源：SQLite

## 一、总判断

吴老师的文档最有价值的部分，不是具体技术栈、目录或视觉风格，而是四个系统原则：创造者主权、真实闭环、角色分离、可复现证据。这些原则值得进入天枢，但必须先完成一次权威转换：文档作者是外部建议者，奈奈才是天枢的创造者、目标拥有者和最终验收者。

因此，不能把四份 Markdown 整体复制为“天枢规则”。正确做法是把原文保留为不可篡改的外部来源，把其中每项主张拆成建议候选，由奈奈逐条采用、调整、延后或拒绝。只有奈奈确认后的结论，才有资格影响正式规则、状态、任务和体验。

天枢的适配公式是：

    奈奈主权 + SQLite 真相 + AgentHub 交互入口 + 天枢判断与治理
    + 有边界执行 + 独立验证 + 奈奈最终验收 + Obsidian 可读工作台

## 二、天枢真正要解决的问题

天枢不是“更聪明的聊天机器人”，也不是把笔记、任务和多个 Agent 放进同一个界面。奈奈真正需要的是一个长期认识她、能形成判断、可以行动但不能越权、会从结果中学习的个人认知与行动操作系统。

完整闭环应当是：

    输入与上下文识别
    -> 事实、推断、不确定项分离
    -> 可解释判断
    -> 奈奈确认或纠正
    -> 有边界执行
    -> Executor 只报告产出
    -> 独立 Verifier 验证证据
    -> 奈奈最终接受或拒绝
    -> 结果复盘
    -> 经验候选
    -> 奈奈确认后版本化
    -> 后续判断明确引用

这个闭环决定了系统边界：聊天内容不是正式记忆，Markdown 不是机器状态，执行完成不等于目标完成，验证通过也不等于奈奈接受。

## 三、为什么需要 AgentHub

AgentHub 的必要性来自“统一入口与协作调度”，不是来自“天枢必须多 Agent 化”。

奈奈的输入会来自对话、手机、语音、文件、Obsidian 和未来设备；执行又可能跨 Codex、浏览器、飞书和其他工具。没有 AgentHub，每个入口都容易形成自己的上下文、状态和结果，奈奈需要反复解释，任务也无法连续恢复。AgentHub 应承担五项职责：

1. 接收不同入口的原始输入，并保留来源与时间。
2. 把天枢生成的判断卡、确认卡和结果卡展示给奈奈。
3. 对异步、跨工具、多步骤工作建立统一运行入口。
4. 展示每个 Agent 正在做什么、做到哪一步、产出了什么。
5. 把结果送回天枢治理层，而不是自行宣布目标完成。

AgentHub 不承担五项职责：

1. 不保存机器正式状态，SQLite 才是唯一真相源。
2. 不替天枢进行核心判断。
3. 不替奈奈确认状态、批准边界或最终验收。
4. 不让 Executor 自证成功。
5. 不把聊天记录自动升级为长期记忆。

并非每个问题都需要多 Agent。简单查询、状态读取、轻量分析可以由天枢直接回答；只有异步、跨工具、多步骤、可并行、涉及外部动作或需要独立验证的工作，才进入 AgentHub 调度链。这样 AgentHub 是必要的工作入口，但不会变成多余的复杂度中心。

## 五、四份文档应该进入哪个系统层

原始文档属于“外部来源层”。它们必须保留文件标识、作者、路径、内容哈希和导入时间，且不可覆盖修改。

拆解出的十三条主张属于“建议候选层”。候选包含原始主张、天枢评估、建议处置、适配方案、优先级和来源引用。

奈奈的采用、调整、延后或拒绝属于“审议决定层”。决定必须记录奈奈身份、理由、最终适配内容和时间，并且追加式保存。

只有已经被奈奈确认的内容，才可以形成“正式规则或变更候选”。正式规则仍需通过对应领域的测试和证据门禁，不能因为完成审议就直接改写运行时。

这一分层解决了过去最危险的混淆：外部文档中的强势表述，不再自动拥有系统权威；天枢自己的分析，也不会伪装成奈奈已经决定。

## 六、Obsidian 为什么显得乱

截图中的根目录同时混合了五种维度：入口导航、生活与项目内容、系统技术层、流程状态、临时队列。例如“入口导航”是导航方式，“项目记忆层”是技术结构，“待处理”是流程状态，“历史归档”是生命周期，而“长期工作区”与“项目推进区”又是内容范围。不同维度并列后，奈奈每次都需要先判断“该按内容找、按状态找、还是按系统层找”，认知成本会持续增加。

更重要的是，“待处理”和“当前状态”不应该依靠手工移动 Markdown 来维护。它们是 SQLite 查询结果：今天有哪些待确认、正在执行、等待验证和等待奈奈验收，应该由 Today 驾驶舱实时生成。

建议的人类可见骨架是：

    00_从这里开始_天枢工作台.md
    10_奈奈/
    20_项目/
    30_生活与关系/
    40_知识与判断/
    50_进化/
    60_活动/
    99_系统/

其中：

- 00 是唯一日常入口，只展示今天焦点、待奈奈决定、进行中、刚学到和需要追问。
- 10 存放奈奈主动确认过的稳定偏好、边界、长期方向和个人状态读模型。
- 20 按真实项目组织，每个项目同时呈现当前状态、时间线、决策、证据和下一步。
- 30 用于生活、家人、关系、健康、出行等个人领域，不与项目技术层混排。
- 40 放知识来源、判断记录、外部建议及其审议结果。
- 50 放问题案例、经验候选、已确认经验、版本变化和后续引用。
- 60 是按时间组织的活动与工作记录，只承担追溯，不承担正式状态。
- 99 放系统说明、字段字典、权限矩阵、生成清单和运维证据，平时不作为工作入口。

原有“06_项目记忆层”和“07_资产索引层”不再作为奈奈需要手工进入的一级目录；它们应变成每个项目页里的自动生成时间线和证据索引。“90_待处理”变成 Today 查询，“98_历史归档”变成按状态和时间筛选的历史视图。

迁移必须非破坏：先在当前隔离仓库生成新读模型，核对文件数、哈希、链接和 SQLite 来源，再由奈奈验收；验收前不移动、不删除、不覆盖原 Obsidian 内容。

## 七、实施顺序

阶段一：权威模型与迁移门禁。明确奈奈、AgentHub、Orchestrator、Executor、Verifier、Obsidian 和外部建议者的能力边界。

阶段二：输入工作空间分类。任何输入先判断属于项目、生活、知识、系统或待奈奈确认，分类本身持久化且可纠正。

阶段三：判断、反馈、结果与经验闭环。把事实、推断、不确定项、奈奈纠正、结果和经验引用连起来。

阶段四：外部建议治理。把四份文档和十三条主张导入 SQLite，提供逐条审议接口和 Today 待确认入口。

阶段五：Obsidian 读模型重构。在隔离目录生成新骨架与映射，不接管、不改写原知识库。

阶段六：Today 驾驶舱与 AgentHub 契约。让奈奈第一眼看见当前唯一焦点、等待她的决定、执行状态、验证结果和新经验。

阶段七：恢复、权限、全回归和视觉验收。证明重启后状态可恢复、越权操作被拒绝、Executor 无法自验、Markdown 无法反向覆盖 SQLite，最后由奈奈验收。

## 八、完成标准

这一轮优化完成，不以“目录变整齐”或“界面更好看”为标准，而以奈奈能否连续完成以下体验为标准：

1. 奈奈给出一段真实输入，天枢能说明它理解了什么、依据是什么、还不确定什么。
2. 奈奈可以纠正判断，纠正被保存且后续可引用。
3. 需要执行时，奈奈能看清边界、执行者、验证者、风险和预期产出。
4. Executor 完成后只能报告产出，独立 Verifier 给出证据结论。
5. 只有奈奈接受后，目标才完成。
6. 结果形成经验候选，奈奈确认后才成为版本化经验。
7. 后续判断能指出引用了哪条已确认经验，以及它是否仍然有效。
8. Obsidian 只展示 SQLite 派生的可读结果，任何 Markdown 编辑都不能悄悄改变机器正式状态。

这才是吴老师文档真正值得天枢学习的部分：不是复制一套话语和目录，而是把创造者主权、真实闭环和工程证据变成奈奈每天可感知、可控制、可纠正的系统行为。
