# 天枢产品闭环与 P5 当前状态报告 002

更新时间：2026-07-15
适用仓库：`liaoj0330-bot/tianshu-next-`
性质：实现说明、验收证据与剩余边界；不是产品完成声明。

## 1. 这次更新解决什么

早期天枢已有可信内核、动态状态、运行治理和统一 Agent 调度，但用户入口仍容易退化为“识别路由标签”或“展示后台”。奈奈真正需要看到的是：今天最重要的事、为什么这样判断、哪些变化需要确认、行动会产生什么后果，以及什么时候 Agent 才能执行。

本轮把底层能力收拢成真实产品路径，同时不越过 P5 尚未完成的验收边界。

## 2. 第一原则

1. 持续接收事实、情绪、变化、问题和行动意图。
2. 用长期身份、当前状态、项目组合和正式证据理解输入。
3. 证据足够时给判断；不足时只问一个关键问题。
4. 可能改变正式状态的内容先成为候选。
5. 行动先形成可理解、可修改、可拒绝的计划。
6. 计划成立不代表可执行；执行者、复核者、路径、超时和重试必须再次确认。
7. Executor 输出只是证据来源，完成结论必须来自独立验证链路。

这对应“奈奈做关键确认，系统主动理解并受控行动”，而不是把编排复杂度推回给用户。

## 3. 已实现的产品闭环

### 3.1 SQLite 今日读模型

`GET /v1/today` 从 SQLite 聚合今日最高优先项目、量化理由、已接受变化、待确认项、正式项目排序、`no_access` 保护项目、最近输入、执行摘要和一个最值得补充的问题。空数据库返回真实空状态，不注入演示项目。

### 3.2 有依据的回答

`direct_answer` 返回 `judgment`、`rationale`、`uncertainty`、`next_action` 和结构化 `evidence`，不再只返回模式名称。

### 3.3 项目组合与量化

项目基线和评估写入 SQLite。量化维度包括使命一致性、系统资产复用、时间窗口、证据质量、依赖紧迫性与资源压力。通用行业词不能自动猜项目；`no_access` 项目即使被明确提到也不能成为执行目标。录音中的澳大利亚内容只是举例，已经移除硬编码规则。

### 3.4 状态候选

明确表达的疲惫或精力不足可以生成当前状态候选，但不会直接覆盖正式状态。确认卡展示变化前后、影响和确认后果，并明确不会因此创建任务或派发 Agent。

### 3.5 计划版本与第一次确认

行动输入生成目标、完成标准、范围、非目标、所需证据、步骤和风险。修改会创建新版本，旧版本进入 `superseded`。第一次确认只创建 Goal、Plan 和待配置 execution boundary，返回 `prepared_not_approved`；Task 与 Run 均不启动。

### 3.6 执行边界与第二次确认

执行边界要求：不同的 executor/verifier、天枢工作区内的 allowed paths、合法 timeout，以及 1–3 次尝试。第二次确认返回 `execution_approved_not_started`：可以创建 Task，但仍不会偷偷启动 Run。

## 4. AgentHub 集成边界

AgentHub 在独立本地仓库，本次提交不复制其源码。天枢仓库只维护稳定 API 和契约。

本地开发版已经验证：今日重点、自然语言入口、项目量化图、状态确认卡、计划修改、两次确认和执行边界表单。1280/1024/768 三档无横向溢出；600 字中文可换行；真实断服务显示中文提示，恢复后重新读取 SQLite。

这证明集成面可用，不等于 P5 真实执行链路完成。

- 空状态不生成演示数据。

## 7. 下一验收门

必须在天枢仓库内完成以下隔离任务：

1. 正常修改：Executor 产生限定变更，独立 verifier 用结构化证据复核。
2. 假成功：退出码 0 但无修改，系统判失败。
3. 越界：未批准路径变化，失败并保留证据。
4. 超时与重试：按批准次数重试，耗尽后进入终态。
5. 取消：终止执行，不产生自动重试。
6. 复核不稳定：不能被当成执行成功，也不能由 Executor 自证完成。
7. 用户呈现：AgentHub 说明发生了什么、证据在哪里、奈奈要做什么决定。

只有真实链路稳定通过，P5 才能从“机制具备”提升为“真实隔离任务验收完成”。
